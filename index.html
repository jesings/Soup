Naive: Procedurally go through all of the cells, and procedurally go through every value for each cell, backtracking if there is no possible value in that cell
1st Optimization: Same as naive, however on every round check(only once) for cells that only have one possible value, and fill those in--I think the reason that this one takes longer than the previous one is because the Naive algorithm never copies the board, it just reverts the changes
2nd Optimization: Same as first optimization, however added checking until there are no further additions to make, added another heuristic that checks if within a clique there is only one cell that could possibly have a certain value, and fills it in. It repeats these 2 heuristics until nothing else can be filled in. Finally, the algorithm no longer fills cells in in order, instead opting to fill in cells with fewer than 3 possibilities first. The fact that the first few are slower is due to the heavy overhead of checking the board over and over again. The majority of the speedum is algorithmic, however code optimizations also certainly played a significant part in cutting down the amount of time each puzzle takes-these optimizations include unrolling loops, lazy evaluation, and simplifying arithmetic.
